# AI活用による知見メモ
2008年頃の設計規約を現代（2026年）の技術スタックにアップデートする過程でAIを活用した。
本書はAIを活用した設計方法論および意思決定プロセスを記録したものである。
今後のAIとの協働開発を円滑に行うための手がかりとなることを期待する。

## 1. 役割分担とワークフロー

### 各モデルの役割分担
以下のAIに規約の技術内容・表現をレビューさせ、指摘を取捨選択しながら規約に反映した。(2026年3月時点)

| AI | 主な用途 |
|---|---|
| Claude Sonnet 4.6| メインの対話パートナー。規約の添削・議論・技術検証・英訳 |
| Claude Opus 4.5| 最終レビュー。トークン消費が大きいため利用には注意が必要 |
| OpenAI ChatGPT 5.2 | 技術内容の妥当性検証・セカンドオピニオン |
| Google Gemini 3 Flash | コード補完・基礎英訳。フィードバックループでのコード生成 |
| DeepL web | 英訳の最終調整（Claude/ChatGPTの翻訳に違和感がある箇所のみ） |

* Gemini利用に当たってはIDEとして使用するGoogle Antigravityからのエージェント/チャット機能を使用している。
* Grokは試してみたが、今回の用途にはしっくりこなかったため途中から適用を除外している。
* CopilotはChat GPTのアウトプットに類似していたためChat GPTに一元化した。
* 英訳についてはClaude Sonnet 4.6が最も気に入っており、DeepL webは最終調整に利用した。


### 設計プロセス
本規約の制作において、AIを単なるレビューツールではなく**対話的な設計パートナー**として活用した。

具体的には以下のプロセスで規約を育てた：

1. **旧規約のレビュー**: 2008年頃の旧版規約をベースに、現代的な基準との差分をAIと議論。
2. **ブログ記事の規約化**: 個人ブログに蓄積した実務知見（障害事例・運用Tips等）をAIと対話しながら規約条項に昇華。
3. **条項の議論と深化**: 各規約の優先度（☆）・例外条件・理由の妥当性をAIと往復しながら洗練。
4. **実装検証**: 実際のDDL（`Snippet-CreateTable.sql`など）をAIにレビューさせ、規約との整合性を確認。そのフィードバックが規約改善にも還元された。
5. **英訳**: 日本語版完成後、翻訳方針（DOA用語の統一など）を合意した上でAIが章ごとに翻訳。

## 2. 人間とAIの対立と合意（Design Decisions）

規約策定において、AI（特にGPT系）は一般的・汎用的なベストプラクティスを推奨する傾向があるが、本規約では**DOA（データ中心アプローチ）**を重視しているため、あえてAIの推奨を棄却、あるいは独自ルールを追加した箇所がある。

* **自然キー vs サロゲートキー**: 
  * AIはサロゲートキー（UUID等）を優先しがちだが、本規約では「データの同一性」を重視し、論理設計レベルでの自然キーの重要性を明文化した（`L_PKY001`, `L_PKY004`）。
* **SQL規約の実効性**:
  * `S_008` (EXISTS優先) や `S_009` (結果順序への非依存) など、パフォーマンス差が僅かであっても、データ不整合や予測不能な挙動（Flakyな挙動）を防ぐための「安全性」を優先する判断を行った。
* **独自の安全基準**:
  * AIが明示的に指摘しなかった観点（`S_SEC006`: 開発環境でのPII取り扱い禁止 等）は設計者が独自に追加。

## 3. AI時代の規約の役割

AIによるコード生成が一般化した現代において、規約は「人間が読むもの」から**「AI（エージェント）が従うべき制約（Guardrail）」**としての役割が強まる。
本規約をAIにコンテキストとして与えることで、生成されるDDLやクエリの品質を自動的に担保し、レビューコストを大幅に削減することが、このプロジェクトの最終的な狙いでもある。

## 4. 雑感 -迷いと違和感-
AIが次々に提案してくる内容について、自分が詳しくない内容が詳細に描かれていた場合、つい無批判に採用をしたくなることがある。
しかしAIは無謬ではないし、己の知見をアウトプットするという本書の趣旨に沿った場合、自分が理解していない内容を採用することは利用者に対する侮辱であると考える。
この項については継続してアップデートを行っていく所存である。
